# 需求管理平台后端领域入口

## 业务边界

本领域承接需求管理平台后端能力，包括项目初始化、项目仓库、项目分支、项目索引知识库、需求、Agent 交接资料、MCP、业务事件和统计。

## 当前状态

- 业务模块 `ruoyi-requirement` 已创建，承载需求平台核心后端能力。
- 平台已从独立基础资料菜单收敛为项目初始化主入口；仓库、项目分支和模块知识作为项目内部数据沉淀，旧 CRUD 接口保留兼容但不再作为左侧菜单入口。
- 项目管理具备初始化聚合接口，管理员可通过 `/requirement/project/init/**` 一次性新增或编辑项目、代码仓库和项目分支，并读取模块摘要、索引摘要和初始化检查项。纯后端服务只维护一条后端仓库即可完成仓库初始化。
- 项目分支对需求人员展示中文标签，对索引和开发流程保存真实分支名，并由后端返回 `initInstruction` 初始化指令；旧 `mcpKey` 仅保留兼容。
- 动作 token 统一沉淀在 `req_action_token`，用于项目初始化、后续需求编排和开发执行等 MCP 动作上下文识别；服务端只保存 token 哈希和前缀。
- 本地 agent 读取开发者机器上的仓库，通过 MCP `publish_repository_index` 上传结构化索引结果，优先使用 `actionToken + remoteUrl` 定位项目分支和仓库；平台只保存相对路径、Git 分支、commit、项目分支和影响面，不保存个人本机目录。
- 新需求创建时，后端按所选项目分支解析真实分支，并从目标仓库或每个仓库最新导入批次中推荐页面、接口、数据表、权限和相关文档。
- 新需求只能提交到已初始化完成的项目分支；后端在需求新增和修改时校验分支归属、分支模块知识和该真实分支的仓库索引覆盖，未初始化分支即使绕过前端也不能落入 `req_demand`。
- 新需求必须指定一个启用的开发人员；提交后普通访问、状态流转、MCP 指令生成和资料包回写锁定在需求创建人和该指定开发人员之间，管理员不受参与人限制。
- 用户可见系统名称为“统一需求流转平台”，保留底层 RuoYi 包名和权限框架兼容项，不做高风险整体框架改名。

## 后续维护

- 新增接口、数据库表、SQL 口径或权限标识时，同步补充 `docs/ai-harness/contracts/` 或 `docs/db/`。
- 新增长期有效业务决策时，同步补充 `docs/ai-harness/decisions/`。
- 调整 MCP tools 或项目索引字段时，同步更新接口契约和数据库关系说明。
