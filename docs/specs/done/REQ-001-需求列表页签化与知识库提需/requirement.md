# 需求列表页签化与知识库提需需求说明

## 背景

前端需求表单要从项目分支知识库读取模块并支持新功能提需；后端需要继续兜底需求编号自动生成、分支初始化校验，并让生成的 Agent 交接资料使用新的任务分支命名格式。

## 目标

- 新增需求继续由后端自动生成需求编号，前端新增时不展示编号输入。
- 需求执行包中的任务分支改为 `fix-功能模块-编号-标题`，实际内容需要经过稳定 slug 化，避免生成不适合命令行的分支名。
- 当需求使用知识库模块或新功能名称时，执行包上下文可以得到清晰模块名。
- 分支已有仓库索引证据时，即使暂时没有既有模块知识，也允许提交新功能需求。
- 影响范围字段继续作为后端保存字段，用于执行包和后续编排，但由前端自动关联。

## 范围

本次包含：

- 调整执行包上下文中的任务分支生成规则。
- 支持从人工模块、索引模块或新功能名称解析模块名。
- 后端分支初始化兜底校验不再把既有模块知识作为新功能提需的前置条件。
- 为需求编号自动生成和任务分支格式补服务测试。
- 同步后端契约和模块 harness 文档。

本次不包含：

- 不新增需求状态。
- 不新增左侧菜单或权限点。
- 不调整持久化结构或脚本。
- 不改变索引发布入口。

## 影响范围

- 接口/API：是，需求保存字段兼容，执行包生成内容变化。
- 持久化结构/脚本：否。
- 权限/菜单：否。
- 页面/交互：间接影响前端需求维护页签和 Agent 交接资料页展示。
- 导出/异步/任务：否。

## 契约与数据口径

- `POST /requirement/demand` 不信任前端传入的 `demandNo`，保存时始终生成 `REQ-yyyyMMdd-序号`。
- 执行包上下文 `taskBranch` 生成格式为 `fix-模块slug-编号slug-标题slug`。
- 模块 slug 优先来自需求关联模块名称；当需求未选择既有模块时，使用需求备注承载的新功能名称。
- 影响范围字段仍保存到需求记录，后续生成执行资料时可继续读取。

## 验收标准

- AC-BE-001：新增需求时后端始终覆盖前端传入的需求编号并生成 `REQ-yyyyMMdd-序号`。
- AC-BE-002：执行包生成的 `taskBranch` 使用 `fix-功能模块-编号-标题` 语义，并输出命令行可用的稳定字符串。
- AC-BE-003：执行包上下文能解析知识库模块或新功能名称，不再只依赖人工模块。
- AC-BE-004：已有仓库索引批次但没有既有模块知识的项目分支，可以保存新功能需求；缺少仓库索引批次时仍拒绝提交。
- AC-BE-005：后端契约和模块 harness 已同步，后端单测、构建和文档/harness 检查通过。

## 约束与假设

- 当前需求平台自身建设阶段允许本地维护 spec，不通过平台 Key 回写。
- 任务分支最终必须是 ASCII，中文模块和标题通过拼音不可得时使用稳定哈希兜底。
- 新功能名称由前端通过需求备注兼容提交，避免新增脚本并且不污染项目分支知识库。
