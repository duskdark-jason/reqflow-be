# harness双轨触发机制需求说明

## 背景

当前后端项目已经内置 `ruoyi-requirement/src/main/resources/harness-template/` 作为项目接入初始化模板源。现有流程主要围绕 Reqflow MCP 接入展开；但实际使用中，部分项目尚未接入 MCP，也需要有与 MCP 接入模式基本一致的触发机制、阶段边界、文档交付物和 harness 更新规则。

## 目标

- 建立 MCP 接入模式和本地 Harness 模式的双轨同构流程，确保没有 MCP 时也按需求设计、执行、Review、返修和完成闭环推进。
- 本地 Harness 模式和 MCP 接入模式都必须保留需求设计确认点；多轮需求设计确认前不得生成执行计划、执行报告或 Review 报告。
- 为模板和当前后端项目补充低 token 功能导航，避免每次都阅读大模块或大契约文档。
- 在模板源头加入模块拆分、契约拆分、决策沉淀和短索引维护触发机制。
- 通过脚本门禁和脚本测试防止新接入项目缺少短索引、保留初始化占位或伪造 MCP 回写。
- 同步前端项目 harness，使前端也具备相同的本地/MCP 双轨触发机制、确认点门禁和搜索导航。

## 可行性评估

- 评估结论：可继续设计
- 主要风险：模板源头、当前项目文档和检查脚本需要同步修改，若只改其中一处会导致后续项目接入规则漂移。
- 需需求人补充或调整：无，用户已确认按双轨同构方向实施。
- 是否允许继续生成需求设计：是，当前需求边界明确。

## 范围

本次包含：

- 更新后端 harness 模板源头、下发文件清单和触发说明。
- 更新全局 reqflow-mcp skill 模板和 MCP 项目初始化技能文案。
- 更新检查脚本与脚本测试，覆盖短索引、初始化占位和本地模式回写边界。
- 同步当前后端项目 `docs/ai-harness` 和流程文档，新增功能导航并记录双轨触发规则。
- 同步当前前端项目 `docs/ai-harness`、流程文档、检查脚本和功能导航。

本次不包含：

- 不接入新的 MCP tool。
- 不改数据库结构、菜单权限或业务接口行为。
- 不执行真实项目接入初始化或发布索引。

## 影响范围

- 接口/API：否，不新增或修改 HTTP/MCP tool 入参出参。
- 数据库/SQL：否。
- 权限/菜单：否。
- 页面/交互：否，不改运行态页面，仅同步前端 harness 文档和脚本。
- 导出/异步/任务：否。
- Harness 模板/流程：是，修改项目接入初始化模板、全局 skill 触发文案和本地流程规则。

## 契约与数据口径

- 接口路径和方法：无接口变更。
- 请求参数：无。
- 响应字段：无。
- 数据粒度：模板文件一行代表一个将下发到目标仓库的 harness 文件；短索引一行代表一个可搜索业务能力入口。

## 验收标准

- AC-001：`harness-template` 下发清单包含短索引模板，模板 `harness-index.json` 暴露 `searchMap` 入口，初始化说明要求新项目维护短索引。
- AC-002：无 MCP 的本地 Harness 模式在模板流程中有明确触发信号、阶段边界、交付物和禁止伪造 MCP 回写规则，并与 MCP 接入模式共用 spec、Review 和 harness 更新闭环。
- AC-003：本地 Harness 模式和 MCP 接入模式都有需求设计确认点门禁；`planning` 状态或需求平台需求设计模式下提前出现 `plan.md`、`execution-report.md` 或 `review-report.md` 会被检查脚本拦截。
- AC-004：模板、当前后端项目和当前前端项目都明确模块/契约拆分阈值、决策沉淀触发条件，以及菜单、接口、权限、数据库和核心流程变更时的短索引更新要求。
- AC-005：检查脚本测试覆盖缺少 `search-map.md`、`harness-index.json` 缺少 `searchMap`、模块文档残留 `初始化待补齐`、本地模式伪造 MCP 回写、需求确认前提前生成执行计划等失败场景。
- AC-006：当前后端和前端项目都新增或更新功能导航文档，能够把 harness 模板、双轨流程、MCP/actionToken、项目初始化、需求执行和验证门禁导向对应文档与代码入口。
